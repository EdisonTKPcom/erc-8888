## Abstract

This ERC proposes a standard interface for an on-chain registry that maps human-readable names to deployed smart contract addresses, enriched with semantic versioning, ownership, audit metadata URIs, and lifecycle status. It enables wallets, block explorers, developer tooling, and other protocols to discover, verify, and reason about smart contracts in a permissionless, composable way — functioning as an on-chain "package registry" for the Ethereum application layer.

## Motivation

The Ethereum ecosystem lacks a unified standard for naming, versioning, and tracking the lifecycle of deployed smart contracts. Today, developers and protocols rely on off-chain documentation, block explorer tags, and ad-hoc naming conventions to communicate what a contract does, who owns it, whether it has been audited, and what version it represents. This creates several problems:

- **Discoverability**: There is no standard way for tooling or other contracts to discover the current address of a named contract (e.g. `"MyProtocol:Vault:v2.1.0"`).
- **Upgradability signalling**: Protocols that deploy new versions cannot broadcast the change in a standard, on-chain-queryable form.
- **Audit transparency**: There is no canonical on-chain anchor for audit reports, security notices, or verification hashes.
- **Lifecycle management**: Contracts can become deprecated, paused, or replaced, but no standard surface exists to reflect this.

ERC-1820 solves interface introspection. ERC-6224 solves intra-protocol dependency injection. ENS solves human-readable address resolution for accounts. This ERC fills the gap at the **application-layer contract registry** level: a neutral, permissionless, and composable index of named, versioned, audited deployments.

The number 8888 is intentionally chosen: it represents prosperity and abundance in Chinese numerology, symbolising the long-term goal of building a richer, more transparent contract ecosystem.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Definitions

- **Name**: A non-empty UTF-8 string used as the primary key for a registry entry (e.g. `"Uniswap:UniversalRouter:v3.0.0"`).
- **Entry**: A struct containing the contract address, owner, version, metadata URI, status, and timestamps.
- **Owner**: The Ethereum address that controls updates to a given registry entry.
- **Status**: An enum reflecting the lifecycle phase of a registered contract.
- **Metadata URI**: A URI resolving to a JSON document conforming to the [Metadata Schema](#metadata-uri-schema) defined in this ERC.

### Status Enum

```solidity
enum ContractStatus {
    Active,       // 0 — deployed and operational
    Deprecated,   // 1 — superseded; use a newer entry
    Paused,       // 2 — temporarily disabled
    Vulnerable,   // 3 — known security issue; do not use
    Decommissioned // 4 — permanently retired
}
```

### Entry Struct

```solidity
struct ContractEntry {
    address contractAddress;   // The deployed contract address
    address owner;             // Address authorised to update this entry
    string  version;           // Semantic version string e.g. "2.1.0"
    string  metadataURI;       // URI to JSON metadata document
    ContractStatus status;     // Current lifecycle status
    uint256 registeredAt;      // block.timestamp at registration
    uint256 updatedAt;         // block.timestamp of last update
}
```

### Interface

Every compliant registry MUST implement `IERC8888` and MUST return `true` for `supportsInterface(0x????????)` (interface ID calculated from the function selectors below).

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.20;

interface IERC8888 {
    // ─── Events ────────────────────────────────────────────────────────────

    /// @notice Emitted when a new entry is registered.
    event ContractRegistered(
        string indexed nameHash,
        string name,
        address indexed contractAddress,
        address indexed owner,
        string version
    );

    /// @notice Emitted when an existing entry is updated.
    event ContractUpdated(
        string indexed nameHash,
        string name,
        address indexed contractAddress,
        ContractStatus status
    );

    /// @notice Emitted when ownership of an entry is transferred.
    event OwnershipTransferred(
        string indexed nameHash,
        string name,
        address indexed previousOwner,
        address indexed newOwner
    );

    // ─── Write Functions ───────────────────────────────────────────────────

    /// @notice Register a new named contract entry.
    /// @dev MUST revert if `name` is already registered.
    ///      MUST revert if `contractAddress` is the zero address.
    ///      MUST revert if `name` is an empty string.
    ///      The caller becomes the entry owner.
    /// @param name           Human-readable name (MUST be unique in this registry).
    /// @param contractAddress Address of the deployed contract.
    /// @param version        Semantic version string.
    /// @param metadataURI    URI resolving to a JSON metadata document.
    function register(
        string calldata name,
        address contractAddress,
        string calldata version,
        string calldata metadataURI
    ) external;

    /// @notice Update an existing entry's address, version, metadata, or status.
    /// @dev MUST revert if the caller is not the entry owner.
    ///      MUST revert if `name` is not registered.
    function update(
        string calldata name,
        address contractAddress,
        string calldata version,
        string calldata metadataURI,
        ContractStatus status
    ) external;

    /// @notice Transfer ownership of a registry entry to a new address.
    /// @dev MUST revert if the caller is not the current owner.
    ///      MUST revert if `newOwner` is the zero address.
    function transferEntryOwnership(
        string calldata name,
        address newOwner
    ) external;

    // ─── Read Functions ────────────────────────────────────────────────────

    /// @notice Retrieve the full entry for a registered name.
    /// @dev MUST revert if `name` is not registered.
    function getEntry(string calldata name)
        external
        view
        returns (ContractEntry memory);

    /// @notice Resolve a name directly to its current contract address.
    /// @dev MUST revert if `name` is not registered.
    function resolve(string calldata name)
        external
        view
        returns (address contractAddress);

    /// @notice Check whether a name is currently registered.
    function isRegistered(string calldata name)
        external
        view
        returns (bool);

    /// @notice Return the status of a registered contract.
    /// @dev MUST revert if `name` is not registered.
    function statusOf(string calldata name)
        external
        view
        returns (ContractStatus);
}
```

### Metadata URI Schema

The JSON document at `metadataURI` SHOULD conform to the following schema:

```json
{
  "$schema": "https://ercs.ethereum.org/assets/erc-8888/metadata-schema.json",
  "name": "string",
  "description": "string",
  "version": "string",
  "repository": "string (URI)",
  "audit": {
    "reports": [
      {
        "auditor": "string",
        "date": "string (ISO 8601)",
        "uri": "string (URI)",
        "sha256": "string (hex)"
      }
    ]
  },
  "abi": "string (URI or inline JSON array)",
  "sourceVerified": "boolean",
  "tags": ["string"]
}
```

All fields are OPTIONAL but implementers SHOULD populate `name`, `version`, `description`, and `audit.reports` when available.

## Rationale

### String names over `bytes32`

Readability was prioritised over gas efficiency for the name key. Registry entries are expected to be written infrequently (once per deployment lifecycle event) but read frequently by tooling and explorers. A string key (e.g. `"Uniswap:UniversalRouter:v3.0.0"`) is self-describing, reduces human error, and improves UX. Protocol-internal dependency injection with `bytes32` keys is already served by ERC-6224.

### Caller-as-owner model

Ownership defaults to the registering address (EOA or contract). This allows deployer scripts, multisigs, DAOs, and factory contracts to all act as natural entry owners without requiring a separate claim step.

### Single global registry vs. per-protocol registries

This ERC defines an interface, not a singleton. Deployers MAY create their own registry instances scoped to their protocol (analogous to ERC-6224) OR deploy against a shared community registry. The interface ensures cross-registry tooling compatibility.

### No on-chain ABI storage

ABI data is large and expensive. The metadata URI pattern — pointing to IPFS, Arweave, or a verifiable HTTPS source — follows established precedent (ERC-721, ERC-1155) and avoids bloating chain state.

### `Vulnerable` status

The explicit `Vulnerable` status, distinct from `Paused`, enables security researchers, auditors, or governance processes to broadcast a known-bad signal on-chain without the entry owner's cooperation (via a governance-controlled admin flag, which implementations MAY add). This is a deliberate security primitive.

## Backwards Compatibility

This ERC introduces no backwards-incompatible changes. It defines a new interface that does not conflict with ERC-165, ERC-1820, ERC-6224, or ENS. Existing contracts that wish to self-register SHOULD call `register` on a deployed `IERC8888` registry from their constructor or a post-deployment script.

## Security Considerations

- **Name squatting**: Because registration is permissionless, a malicious actor could register well-known names before legitimate owners. Implementations MAY add governance, bonding, or challenge mechanisms. Users SHOULD verify entry ownership independently.
- **Metadata URI trust**: The registry stores a URI pointer, not the metadata content. URIs can be mutated (HTTPS) or return invalid JSON. Tooling SHOULD validate the `sha256` field in audit reports against fetched documents.
- **Status accuracy**: The `Vulnerable` and `Deprecated` statuses are advisory signals; the registry cannot prevent interaction with the underlying contract. Users MUST NOT assume safety solely from status.
- **Owner key compromise**: If an entry owner's key is compromised, the attacker can point the entry to a malicious address. Owners SHOULD use multisigs or timelocked governance contracts.
